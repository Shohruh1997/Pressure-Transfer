# Roadmap и backlog разработки Universal Converter

## 1. Назначение документа

Документ переводит требования из `UNIVERSAL_CONVERTER_TZ.md` в практический план разработки.

Цели документа:

1. Разбить разработку на понятные этапы.
2. Зафиксировать состав MVP.
3. Сформировать backlog задач в приоритетном порядке.
4. Показать зависимости между этапами.
5. Дать основу для оценки сроков и распределения работ.

## 2. Принципы планирования

Разработка должна идти от платформы к категориям, а не наоборот.

Порядок выбран так, чтобы:

1. Сначала создать универсальную архитектуру категорий.
2. Затем реализовать общий движок интерфейса и карточек.
3. После этого подключить физические категории с локальной математикой.
4. Далее добавить валютную категорию как наиболее зависимую от внешних данных.
5. После MVP заняться поиском, избранным, кешем и новыми категориями.

## 3. Целевые релизы

### 3.1. Release 0.1 — Product Foundation

Цель релиза:

- создать каркас единого SPA;
- заложить архитектуру категорий;
- подготовить базовый UI-shell.

### 3.2. Release 0.2 — Core Conversion Engine

Цель релиза:

- запустить общий механизм конвертации;
- реализовать карточки результатов;
- добавить базовые настройки вывода и сохранение состояния.

### 3.3. Release 0.3 — Physical Categories MVP

Цель релиза:

- полноценно реализовать категории `Давление` и `Температура`;
- подтвердить, что архитектура работает для коэффициентных и формульных категорий.

### 3.4. Release 0.4 — Currency MVP

Цель релиза:

- добавить категорию `Валюта`;
- подключить внешние API;
- реализовать актуальные и исторические курсы.

### 3.5. Release 1.0 — MVP Public Ready

Цель релиза:

- довести продукт до состояния публичной публикации;
- закрыть UX, стабильность и приемочные критерии MVP.

### 3.6. Release 1.1+ — Expansion

Цель релиза:

- добавить новые категории;
- усилить UX;
- улучшить офлайн-устойчивость и масштабируемость.

## 4. Этапы разработки

## Этап 0. Подготовка и проектирование

Цель этапа:

- подготовить репозиторий к разработке единого продукта и уточнить технический каркас.

Результат этапа:

- понятная структура фронтенд-проекта;
- схема модулей;
- согласованный формат описания категорий.

Задачи этапа:

1. Выбрать финальную структуру каталогов нового сайта.
2. Решить, будет ли новый продукт жить в корне проекта или в отдельной папке.
3. Зафиксировать формат конфигурации категорий.
4. Зафиксировать формат хранения пользовательских настроек.
5. Подготовить базовый README по запуску нового сайта.

Критерий готовности этапа:

- разработчик может начать реализацию без архитектурной неопределенности.

## Этап 1. Базовый shell приложения

Цель этапа:

- создать единый интерфейсный каркас сайта.

Результат этапа:

- страница продукта открывается как единое SPA;
- есть базовая навигация между категориями;
- есть универсальная зона ввода и зона результатов.

Задачи этапа:

1. Создать общий layout сайта.
2. Реализовать header с названием продукта.
3. Реализовать переключатель темы.
4. Реализовать верхнюю навигацию категорий.
5. Реализовать контейнер текущей категории.
6. Реализовать универсальный контейнер карточек результатов.
7. Подготовить адаптивную верстку для desktop и mobile.

Критерий готовности этапа:

- пользователь может переключаться между заглушками категорий без перезагрузки.

## Этап 2. Универсальный движок категорий

Цель этапа:

- реализовать ядро, через которое будут работать все категории.

Результат этапа:

- каждая категория описывается конфигурацией;
- общий UI работает поверх конфигурации, а не поверх жестко зашитых условий.

Задачи этапа:

1. Реализовать реестр категорий.
2. Реализовать схему объекта категории.
3. Реализовать поддержку разных типов адаптеров конвертации.
4. Реализовать общий рендер списка единиц.
5. Реализовать общий механизм выбора исходной единицы.
6. Реализовать общий механизм рендера карточек результатов.
7. Реализовать единый парсер и форматтер чисел с возможностью переопределения по категории.
8. Реализовать базовое уведомление о копировании.

Критерий готовности этапа:

- можно подключить как минимум одну категорию только через конфигурацию и адаптер.

## Этап 3. Настройки, localStorage и порядок вывода

Цель этапа:

- перенести в новый продукт сильные UX-механики из текущего валютного конвертера.

Результат этапа:

- пользователь управляет активными единицами;
- настройки и порядок карточек сохраняются между сессиями.

Задачи этапа:

1. Реализовать модальное окно настроек.
2. Реализовать чекбоксы активных единиц для текущей категории.
3. Реализовать защиту от пустого набора активных единиц.
4. Реализовать сохранение активной категории.
5. Реализовать сохранение темы.
6. Реализовать сохранение исходной единицы по категории.
7. Реализовать сохранение активных единиц по категории.
8. Реализовать сохранение пользовательского порядка карточек.
9. Реализовать восстановление состояния из `localStorage`.

Критерий готовности этапа:

- после перезагрузки страницы пользователь получает ту же тему, категорию и набор карточек.

## Этап 4. Категория «Давление»

Цель этапа:

- реализовать ключевую категорию, которая отражает главное отличие продукта.

Результат этапа:

- пользователь вводит значение в любой единице давления и сразу видит набор нужных выходных единиц.

Задачи этапа:

1. Подготовить конфигурацию единиц давления.
2. Вынести коэффициенты давления в централизованный реестр.
3. Реализовать адаптер коэффициентной конвертации через базовую единицу Па.
4. Реализовать значения по умолчанию для категории давления.
5. Реализовать корректное форматирование больших и малых значений.
6. Реализовать копирование результата из карточки.
7. Проверить корректность формул на эталонных значениях.

Критерий готовности этапа:

- сценарий `Па -> бар, атм, кгс/см², мм рт. ст.` работает сразу и без выбора каждой пары вручную.

## Этап 5. Категория «Температура»

Цель этапа:

- проверить, что продукт поддерживает не только коэффициентные, но и формульные категории.

Результат этапа:

- пользователь может вводить значение в Цельсиях, Фаренгейтах или Кельвинах и сразу видеть остальные шкалы.

Задачи этапа:

1. Подготовить конфигурацию температурных единиц.
2. Реализовать отдельный адаптер температурных формул.
3. Выбрать каноническую внутреннюю шкалу.
4. Реализовать корректные преобразования между `C`, `F`, `K`.
5. Настроить вывод карточек по тому же общему UI.
6. Проверить эталонные точки: `0°C`, `100°C`, `32°F`, `273.15K`.

Критерий готовности этапа:

- температурная категория работает на том же UI-движке, что и давление.

## Этап 6. Категория «Валюта»

Цель этапа:

- реализовать самый сложный модуль MVP с внешними данными и датой.

Результат этапа:

- пользователь получает актуальные и исторические курсы в общем интерфейсе Universal Converter.

Задачи этапа:

1. Подготовить конфигурацию поддерживаемых валют.
2. Реализовать валютный data provider.
3. Подключить основной источник актуальных курсов.
4. Подключить fallback-источник.
5. Реализовать выбор даты.
6. Реализовать загрузку курсов на дату.
7. Реализовать блокировку UI на время загрузки.
8. Реализовать отображение ошибки загрузки.
9. Реализовать пересчет через базовую валюту USD.
10. Сохранить настройки валют по модели текущего `course-convertor`.

Критерий готовности этапа:

- пользователь может выбрать дату, ввести сумму и сразу получить результаты по активным валютам.

Результат этапа:

## 5. Зависимости между этапами

Критические зависимости:

1. Этап 1 зависит от завершения этапа 0.
2. Этап 2 зависит от этапа 1.
3. Этап 3 зависит от этапа 2.
4. Этапы 4 и 5 зависят от этапов 2 и 3.
5. Этап 6 зависит от этапов 2 и 3, но не зависит от полного завершения этапов 4 и 5.
6. Этап 7 зависит от завершения этапов 4, 5 и 6.
7. Этап 8 должен начинаться только после стабилизации MVP.

Практический вывод:

- давление и температура можно вести параллельно после готовности ядра;
- валюту лучше брать отдельным этапом после стабилизации архитектуры, так как она добавляет сетевые риски.

## 6. Приоритеты backlog

Обозначения приоритета:

- `P0` — обязательно для старта разработки или блокирует другие задачи.
- `P1` — обязательно для MVP.
- `P2` — важно, но можно завершать после основной функциональности.
- `P3` — вторая очередь или улучшения.

## 12. Риски проекта

Основные риски:

1. Слишком ранняя реализация валютной категории до стабилизации ядра может привести к лишнему усложнению архитектуры.
2. Если не сделать конфигурационную архитектуру категорий, проект быстро превратится в набор отдельных конвертеров в одном файле.
3. Если не разделить коэффициентные и формульные категории через адаптеры, расширение продукта станет дорогим.
4. Внешние API валют могут быть нестабильны, поэтому нужен fallback и желательно кеш.
5. Без ранней проверки mobile UX сайт может остаться удобным только на десктопе.

## 13. Definition of Done для MVP

MVP считается завершенным, если:

1. Есть единый сайт с категориями `Давление`, `Температура`, `Валюта`.
2. Для каждой категории работает выбор исходной единицы.
3. Для каждой категории работает вывод нескольких настраиваемых карточек результатов.
4. Пользователь может менять набор и порядок карточек.
5. Настройки сохраняются в `localStorage`.
6. Поддерживаются светлая и темная тема.
7. В валютной категории работают актуальные и исторические курсы.
8. Сайт пригоден для публикации на статическом хостинге.

## 14. Рекомендуемый следующий практический шаг

Следующим прикладным шагом разработки должен стать старт `Этапа 0` и `Этапа 1` одновременно:

1. выделить каталог нового сайта;
2. создать базовый SPA-shell;
3. сразу заложить реестр категорий;
4. не начинать валютную категорию до готовности общего ядра.
