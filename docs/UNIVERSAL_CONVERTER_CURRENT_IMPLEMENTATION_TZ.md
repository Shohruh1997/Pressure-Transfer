# Техническое задание на текущую реализацию Universal Converter

## 1. Назначение документа

Этот документ фиксирует текущее фактическое состояние корневого SPA `Universal Converter` и служит единым источником истины по:

- структуре файлов;
- пользовательскому интерфейсу;
- логике категорий;
- составу кнопок и действий;
- состоянию приложения и `localStorage`;
- ограничениям текущей реализации.

Документ нужен потому, что существующие файлы `PROJECT_TZ.md`, `UNIVERSAL_CONVERTER_TZ.md` и `UNIVERSAL_CONVERTER_STRUCTURE_TZ.md` полезны как продуктовые и архитектурные ориентиры, но не описывают полностью текущую реализацию в одном месте и частично расходятся с текущим кодом.

## 2. Статус документа

- Редакция: актуальное состояние репозитория на 24.05.2026.
- Объект описания: корневой статический SPA `Universal Converter`.
- Формат запуска: без сборки, через локальный HTTP-сервер.

## 3. Точка входа и запуск

### 3.1. Runtime-модель

Приложение является статическим сайтом без собственного бэкенда и без npm-сборки.

### 3.2. Точка входа

- HTML-вход: `index.html`
- JS-вход: `src/main.js`

### 3.3. Локальный запуск

Проект должен запускаться через локальный HTTP-сервер. Для текущей локальной проверки использовался адрес `http://127.0.0.1:4173/`.

## 6. Глобальная структура интерфейса

Приложение состоит из одного общего shell.

### 6.1. Верхняя зона

В верхней части интерфейса находятся:

1. Название продукта `Universal Converter`.
2. Заголовок `Конвертер величин`.
3. Краткое описание активной категории.
4. Переключатель темы.
5. Навигация по категориям.
6. Кнопка `Калькулятор` для валютной категории.

### 6.2. Основной layout

Основной layout состоит из двух зон:

1. `inputPanel`.
2. `resultsPanel`.

Поведение layout зависит от категории:

- для `Температура` используется inline-режим, где список температурных полей рендерится внутри `inputPanel`, а `resultsPanel` скрыт;
- для `Давление` используется inline-режим, где список полей рендерится внутри `inputPanel`, а `resultsPanel` скрыт;
- для `Валюта` используется inline-режим, а `resultsPanel` показывается только для встроенного калькулятора.
- для `Длина`, `Вес`, `Площадь`, `Объем`, `Скорость`, `Время` и `Энергия` используется общий factor-режим: одно поле ввода, выбор исходной единицы и набор result-card по остальным единицам.

### 6.3. Модальное окно

В приложении используется одна общая модалка `currencySettingsModal`, которая переиспользуется для:

- выбора валют;
- выбора и порядка единиц давления;
- выбора и порядка шкал температуры.

Заголовок и тело модалки зависят от активной категории.

## 8. Категория «Давление»

### 8.1. Формат UI

Категория давления реализована не через общий `input + select + result cards`, а в том же inline-стиле, что и валюта.

Внутри блока давления находятся:

1. строка служебных чипов в шапке блока;
2. кнопка-шестеренка открытия настроек;
3. список активных полей давления;
4. нижняя кнопка `Настройки`;
5. кнопка очистки и кнопка поднятия вверх внутри каждой строки.

### 8.2. Поддерживаемые единицы

| Код | Короткая подпись | Полное название | Коэффициент к Па |
| --- | --- | --- | --- |
| `pa` | `Па` | Паскаль | `1` |
| `kpa` | `кПа` | Килопаскаль | `1000` |
| `mpa` | `МПа` | Мегапаскаль | `1000000` |
| `bar` | `бар` | Бар | `100000` |
| `kgfcm2` | `кгс/см²` | кгс/см² | `98066.5` |
| `atm` | `атм` | Атмосфера | `101325` |
| `mmhg` | `мм рт. ст.` | Миллиметр ртутного столба | `133.322368` |

### 8.3. Математическая модель

Внутренняя расчетная база для давления: Паскаль.

Формулы:

```text
valueInPa = inputValue * sourceUnit.factor
targetValue = valueInPa / targetUnit.factor
```

### 8.4. Поведение ввода

Пользователь может вводить значение в любой активной строке давления.

При вводе происходит:

1. фиксация активной единицы в `pressureLastActiveUnitId`;
2. сохранение чернового отображаемого значения в `pressureDraftValue`;
3. пересчет `pressureBasePaAmount`;
4. обновление остальных активных полей без полного перерендера контейнера.

При фокусе:

- значение выделяется целиком, чтобы новый ввод сразу заменял старое.

При очистке:

- обнуляется расчетная база категории;
- поле остается в фокусе;
- кнопку очистки скрывают и отключают.

### 8.5. Настройки давления

Настройки давления работают по тому же паттерну, что и у валют.

Механика:

1. открытие по кнопке-шестеренке в шапке;
2. открытие по нижней кнопке `Настройки`;
3. в модалке отображаются все единицы давления;
4. активные единицы отмечены чекбоксами;
5. неактивные единицы отображаются ниже активных;
6. drag-and-drop доступен только для активных единиц;
7. нельзя отключить последнюю активную единицу;
8. порядок активных единиц сохраняется;
9. состав активных единиц сохраняется.

### 8.6. Кнопки давления

| Кнопка | Где находится | Назначение |
| --- | --- | --- |
| `⚙` | В шапке pressure-блока | Открыть модалку выбора единиц давления |
| `×` | В строке единицы | Очистить значение текущей единицы |
| `↑` | В строке единицы | Поднять единицу наверх списка |
| `Настройки` | Внизу pressure-блока | Открыть ту же модалку настроек |
| `×` в модалке | Заголовок модалки | Закрыть модалку |

### 8.7. Состояние давления

Для давления используются поля состояния:

- `pressureBasePaAmount`
- `pressureDraftValue`
- `pressureLastActiveUnitId`
- `pressureUnitsOrder`

Поле `pressureUnitsOrder` в текущей реализации означает именно список активных и видимых единиц в пользовательском порядке.

## 9. Категория «Валюта»

### 9.1. Формат UI

Валюта реализована как inline-список активных валют с отдельным встроенным калькулятором.

Внутри блока валют находятся:

1. чип даты;
2. чип базовой валюты `USD`;
3. кнопка-шестеренка открытия настроек;
4. список активных валютных строк;
5. нижняя кнопка `Настройки`.

### 9.2. Поддерживаемые валюты

Поддерживаемый список валют:

- `RUB`
- `UZS`
- `KZT`
- `USD`
- `EUR`
- `GBP`
- `TRY`
- `CNY`
- `JPY`
- `AED`

### 9.3. Расчетная модель

Внутренняя расчетная база: USD.

Формулы:

```text
amountInUsd = inputAmount / sourceRate
targetAmount = amountInUsd * targetRate
```

### 9.4. Источник курсов в текущей реализации

В текущем корневом SPA используются локальные статические курсы из `src/config/app-config.js`.

Это означает:

- дата в шапке сейчас является статическим визуальным элементом;
- загрузка live API в текущей корневой реализации не подключена;
- текущая логика валюты является локальной offline-моделью.

### 9.5. Настройки валют

В модалке валют доступны:

1. чекбоксы включения и выключения валют;
2. reorder активных валют drag-and-drop;
3. защита от пустого набора активных валют.

## 10. Категория «Температура»

### 10.1. Формат UI

Температура реализована в том же inline-стиле, что и `Валюта` и `Давление`.

Внутри блока температуры находятся:

5. кнопка очистки и кнопка поднятия вверх внутри каждой строки.

### 10.2. Поддерживаемые шкалы

- `°C`
- `°F`
- `K`

### 10.3. Математическая модель

Внутренняя универсальная модель реализована через преобразование в Цельсий.

Формулы:

```text
F -> C = (F - 32) * 5 / 9
K -> C = K - 273.15
C -> F = C * 9 / 5 + 32
C -> K = C + 273.15
```

### 10.4. Поведение ввода

Пользователь может вводить значение в любой активной строке температуры.

При вводе происходит:

2. сохранение чернового отображаемого значения в `temperatureDraftValue`;
3. пересчет `temperatureBaseCelsiusAmount`;
4. обновление остальных активных полей без полного перерендера контейнера.

При фокусе:

- значение выделяется целиком, чтобы новый ввод сразу заменял старое.

При очистке:

- обнуляется расчетная база категории;
- поле остается в фокусе;
- кнопку очистки скрывают и отключают.

### 10.5. Настройки температуры

3. в модалке отображаются все температурные шкалы;

### 10.8. Дополнительные реализованные factor-категории

Ниже перечислены дополнительные категории, которые уже заведены в runtime и работают через общий коэффициентный движок `factor`.

Для всех этих категорий в текущей реализации используется единый UI-паттерн:

1. одно поле ввода;
2. один select исходной единицы;
3. автоматический пересчет без кнопки запуска;
4. result-card по остальным единицам;
5. кнопка копирования на каждой карточке.

1. отдельной settings-modal;

Они уже работают как полноценные категории конвертации, но используют общий базовый UI общего factor-движка.

## 15. Правила форматирования и ввода

### 15.1. Парсинг

Для пользовательского ввода используются следующие правила:

- пробелы в числе удаляются;
- запятая нормализуется как десятичный разделитель;
- промежуточные draft-значения допускают формат наподобие `1,`.

### 15.2. Форматирование

Используются две основные стратегии:

1. `formatNumber()` для инженерных и расчетных значений;
2. `formatCurrencyInputDraft()` и связанное форматирование для inline-полей ввода.

### 15.3. Copy-to-clipboard

Копирование использует `navigator.clipboard`, а при недоступности переходит на fallback через временный `textarea`.

## 16. Текущие ограничения и честные границы реализации

### 16.1. Что уже есть

- валюта, давление и температура работают как inline-многопольные категории;
- длина, вес, площадь, объем, скорость, время и энергия работают как отдельные категории на общем factor-движке;
- для давления и валют есть модалка настроек;
- для температуры тоже есть модалка настроек;
- есть drag-and-drop reorder;
- есть очистка полей;
- есть тема;
- есть сохранение состояния в `localStorage`.

### 16.2. Что еще не реализовано в текущем root SPA

- live API валютных курсов;
- реальный выбор даты курсов;
- отдельный внешний CSS-файл вместо встроенных стилей `index.html`;
- автоматизированные тесты.

## 17. Критерии актуальности документа

Документ считается актуальным, если после изменений в коде поддерживается соответствие по следующим точкам:

1. список файлов и модулей соответствует фактической структуре `src`;
2. список категорий соответствует `categoryRegistry`;
3. список кнопок соответствует реальному DOM и обработчикам в `main.js`;
4. таблицы единиц и валют соответствуют `app-config.js`;
5. ограничения документа не обещают того, чего в коде нет.
