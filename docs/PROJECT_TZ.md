# Техническое задание на проект Pressure-Transfer Workspace

## 1. Статус документа

- Документ подготовлен по результатам анализа содержимого репозитория на 16.05.2026.
- Основание: фактическая структура каталогов, README, HTML/CSS/JavaScript, C++/Qt-код, UI-формы, qmake-проекты и размещенные бинарные артефакты.
- Документ описывает текущее состояние системы и формализует требования к ее сопровождению, доработке и публикации.
- Актуальный боевой URL основной веб-версии проекта: `https://flex-convert.ecomdev.uz/`.

## 2. Объект ТЗ

Объектом ТЗ является не одно приложение, а workspace из нескольких связанных по тематике модулей:

1. Веб-конвертер давления в корне проекта.
2. Веб-конвертер валют в каталоге `course-convertor`.
3. Набор legacy desktop-приложений на Qt/C++ в каталоге `legacy_cpp`.

В рамках настоящего ТЗ проект трактуется как пакет прикладных конвертеров и вычислительных утилит, в котором веб-модули являются актуальной пользовательской веткой, а Qt-модули выступают как унаследованные desktop-решения и справочный прототип предметной логики.

## 3. Цель проекта

Создать и сопровождать набор легковесных приложений для быстрого пересчета числовых значений в инженерных и финансовых сценариях без обязательного серверного бэкенда.

Цели проекта:

- обеспечить быстрый и понятный пересчет единиц давления;
- обеспечить быстрый мультивалютный пересчет по актуальным и историческим курсам;
- сохранить и при необходимости поддерживать legacy desktop-решения на Qt;
- минимизировать вычислительные ошибки за счет использования единой математической модели и прозрачных коэффициентов конвертации;
- предоставить решение, которое можно запускать локально, публиковать на статическом хостинге и при необходимости использовать как офлайн-утилиту в desktop-формате.

## 5. Пользователи и сценарии использования

Основные категории пользователей:

- инженер, техник или студент, которому нужен быстрый пересчет давления;
- пользователь, которому требуется мгновенный пересчет валют без сложного интерфейса;
- разработчик или владелец проекта, поддерживающий текущую веб-ветку и legacy-кодовую базу;
- пользователь Windows, запускающий подготовленные Qt `.exe`-файлы без самостоятельной сборки.

Ключевые сценарии:

1. Пользователь открывает веб-конвертер давления, вводит число и сразу получает все остальные единицы.
2. Пользователь копирует рассчитанное значение давления в буфер обмена.
3. Пользователь открывает конвертер валют, вводит сумму в одной валюте и получает пересчет во всех активных валютах.
4. Пользователь выбирает историческую дату и получает пересчет по курсам на выбранную дату.
5. Пользователь добавляет или убирает валюты из списка и сохраняет предпочтения в браузере.
6. Пользователь Windows запускает legacy-приложения без ручной настройки среды Qt.

## 6. Границы поставки

В поставку входят:

- исходные коды всех веб- и desktop-модулей;
- пользовательские интерфейсы на HTML и Qt Designer `.ui`;
- qmake-проекты `.pro`;
- бинарные сборки `.exe` и часть runtime-зависимостей Qt/MinGW;
- ресурсные файлы и иконки;
- README-документация;

В поставку не входят:

- серверный API собственной разработки;
- база данных;
- учетные записи пользователей;
- облачная синхронизация настроек;
- автоматизированные тесты и CI/CD-конфигурация.

## 7. Функциональные требования

### 7.1. Pressure Converter Web

Модуль размещен в корне проекта, использует `index.html` как HTML-вход и модульную JavaScript-структуру в каталоге `src`.

Функциональные требования:

1. Приложение должно открываться как статическая веб-страница без шага сборки.
2. На экране должен присутствовать один основной ввод числового значения.
3. Значение по умолчанию при загрузке страницы должно быть равно `1`.
4. Пользователь должен иметь возможность выбрать исходную единицу давления из выпадающего списка.
5. Список поддерживаемых единиц должен включать:
   - Па;
   - кПа;
   - МПа;
   - бар;
   - кгс/см²;
   - атм;
   - мм рт. ст.
6. Вычисления должны выполняться по единой схеме через базовую единицу измерения, которой является Паскаль.
7. Формула пересчета должна иметь вид:
   - `valueInPascals = inputValue * sourceFactor`;
   - `targetValue = valueInPascals / targetFactor`.
8. После каждого изменения значения или единицы источник пересчета должен происходить автоматически, без отдельной кнопки расчета.
9. На экране должен выводиться список результатов по всем остальным единицам, кроме выбранной исходной единицы.
10. Для каждого результата должна отображаться кнопка копирования.
11. После успешного копирования должно выводиться toast-уведомление `Скопировано!`.
12. При вводе нечислового значения должно показываться понятное сообщение об ошибке.
13. Числа должны форматироваться для удобного чтения:
   - обычный формат для типовых значений;
   - экспоненциальная запись для очень больших и очень малых величин.
14. Интерфейс должен оставаться пригодным для мобильного использования и на десктопе.
15. Приложение должно работать без бэкенда и без сторонних npm-зависимостей.

Требования к математической модели:

| Единица | Идентификатор | Коэффициент к Па |
| --- | --- | --- |
| Паскаль | `pa` | `1` |
| Килопаскаль | `kpa` | `1000` |
| Мегапаскаль | `mpa` | `1000000` |
| Бар | `bar` | `100000` |
| кгс/см² | `kgf` | `98066.5` |
| Атмосфера физическая | `atm` | `101325` |
| мм рт. ст. | `mmhg` | `133.322368` |

### 7.2. Currency Converter Web

Модуль размещен в каталоге `course-convertor` и реализован как SPA на ES Modules.

Функциональные требования:

1. Приложение должно отображать список активных валютных полей ввода.
2. Пользователь должен иметь возможность вводить сумму в любом активном поле.
3. При вводе значения в одной валюте остальные валюты должны пересчитываться мгновенно.
4. Внутренняя расчетная модель должна использовать USD как базовую валюту.
5. Формулы пересчета должны иметь вид:
   - `amountInUsd = amount / fromRate`;
   - `targetAmount = amountInUsd * toRate`.
6. По умолчанию должны быть активны валюты `RUB`, `UZS`, `KZT`, если в `localStorage` не сохранено другое состояние.
7. Поддерживаемые валюты должны включать:
   - RUB;
   - UZS;
   - KZT;
   - USD;
   - EUR;
   - GBP;
   - TRY;
   - CNY;
   - JPY;
   - AED.
8. Для каждой валюты должны отображаться код и флаг.
9. Пользователь должен иметь возможность изменить порядок валют, поднимая нужную строку наверх отдельной кнопкой.
10. Приложение должно сохранять порядок и состав активных валют в `localStorage`.
11. Пользователь должен иметь возможность открыть модальное окно настроек и включить или отключить валюты чекбоксами.
12. Приложение должно поддерживать выбор даты через календарь.
13. При выборе сегодняшней даты должны загружаться актуальные курсы.
14. При выборе прошлой даты должны загружаться исторические курсы.
15. Приложение должно иметь темную и светлую тему.
16. Тема должна определяться из `localStorage`, а при отсутствии сохранения ориентироваться на системную тему.
17. Пользователь должен иметь возможность вручную переключать тему.
18. Числа должны форматироваться по локали `ru-RU` с пробелами как разделителями тысяч.
19. При временной недоступности основного API актуальных курсов приложение должно использовать fallback-источник.
20. При полной недоступности источников курсов приложение должно выводить сообщение о невозможности загрузки данных.

Требования к интеграциям:

1. Для актуальных курсов должен использоваться публичный API `https://open.er-api.com/v6/latest/USD`.
2. Для исторических курсов и fallback-доступа должен использоваться `@fawazahmed0/currency-api` через:
   - `https://cdn.jsdelivr.net/npm/@fawazahmed0/currency-api@{date}/v1/currencies/usd.json`;
   - `https://{date}.currency-api.pages.dev/v1/currencies/usd.json` или `https://latest.currency-api.pages.dev/v1/currencies/usd.json`.
3. Приложение не должно требовать API-ключа.
4. Для запуска локально должен использоваться HTTP-сервер, так как ES Modules не гарантируют корректную работу при открытии через `file://`.

Требования к сохранению состояния:

1. В `localStorage` должен храниться ключ `activeCurrencies`.
2. В `localStorage` должен храниться ключ `theme`.
3. Значения должны использоваться при повторном открытии страницы.

### 7.3. Engineering Calculator (Legacy Qt)

Модуль реализован на C++/Qt и предназначен для базовых инженерных вычислений в desktop-формате.

Функциональные требования к поддерживаемому поведению:

1. Приложение должно запускаться как окно `QMainWindow`.
2. На экране должен отображаться дисплей для текущего выражения/значения.
3. Должны присутствовать цифровые кнопки `0-9`.
4. Должны присутствовать кнопки арифметических операций:
   - `+`;
   - `-`;
   - `x`;
   - `/`.
5. Должны присутствовать кнопки `=`, `AC`, `del`, `%`, `+/-`, `.`.
6. Кнопка смены знака должна инвертировать знак текущего значения.
7. Кнопка десятичной точки должна позволять ввод дробных чисел.
8. Отображение результата должно поддерживать точность до 16 значащих цифр.

Требования к сопровождению legacy-модуля:

1. Логика доступных кнопок должна полностью соответствовать фактическому интерфейсу.
2. Операции `del`, `%` и `AC` должны быть реализованы явно и предсказуемо.
3. Недоступные или незавершенные возможности не должны отображаться как рабочие.
4. Деление на ноль должно обрабатываться безопасно.

### 7.4. Pressure Converter Qt (Legacy)

Модуль реализован на C++/Qt как desktop-предшественник веб-конвертера давления.

Функциональные требования:

1. Пользователь должен выбрать исходную единицу давления через `QComboBox`.
2. Пользователь должен ввести значение в поле ввода.
3. После нажатия кнопки расчета должны быть заполнены поля результатов по всем поддерживаемым единицам.
4. Поддерживаемые единицы должны соответствовать веб-версии:
   - Па;
   - кПа;
   - МПа;
   - бар;
   - кгс/см²;
   - атм;
   - мм рт. ст.
5. Для сопровождения и исправления legacy-версии расчет обязан быть переписан через базовую единицу Па по той же математической модели, что используется в веб-версии.
6. В интерфейсе не должно быть расхождения между названием единиц и расчетной логикой.

### 7.5. Temperature Converter Qt (Legacy)

Модуль реализован на C++/Qt и предназначен для пересчета температуры.

Функциональные требования:

1. Пользователь должен иметь возможность выбрать исходную шкалу:
   - Цельсий;
   - Фаренгейт;
   - Кельвин.
2. Пользователь должен ввести температуру и выполнить расчет кнопкой.
3. Приложение должно выводить значения по всем трем шкалам одновременно.
4. Должна присутствовать кнопка очистки исходного поля.
5. Должны использоваться стандартные формулы:
   - `K = C + 273.15`;
   - `F = C * 9 / 5 + 32`;
   - `C = (F - 32) * 5 / 9`;
   - `C = K - 273.15`.
6. Выбор исходной шкалы не должен приводить к неоднозначному состоянию внутренних флагов.

## 8. Требования к интерфейсу

Общие требования:

1. Пользовательский интерфейс должен быть понятным без внешней документации.
2. Основные действия должны быть доступны с первого экрана.
3. На веб-экранах интерфейс должен быть адаптивным и пригодным для мобильных устройств.
4. Основные поля ввода должны визуально выделяться.
5. Текстовые подписи единиц, валют и действий должны быть на русском языке, кроме международных валютных кодов.

Дополнительные требования по модулям:

1. Pressure Converter Web должен визуально имитировать компактный экран калькулятора/конвертера.
2. Currency Converter Web должен включать:
   - боковой блок инструкций;
   - заголовок;
   - поле выбора даты;
   - переключатель темы;
   - список валютных строк;
   - кнопку открытия настроек;
   - модальное окно выбора валют.
3. Desktop-модули должны использовать стандартные элементы Qt Widgets.

## 9. Архитектурные требования

### 9.1. Веб-модули

1. Веб-модули должны работать без серверной бизнес-логики.
2. Для корневого конвертера допускается одна страница HTML с подключенным JS-файлом.
3. Для валютного конвертера логика должна быть разделена как минимум на:
   - сетевой слой;
   - математический слой;
   - слой управления DOM и состоянием.
4. Веб-приложения должны быть совместимы со статическим хостингом.
5. Не допускается зависимость от приватных API или секретов.

### 9.2. Desktop-модули

1. Desktop-модули должны собираться через qmake `.pro`-файлы.
2. Основные модули Qt должны включать `core`, `gui`, `widgets`.
3. Сборка должна быть совместима как минимум с C++11.
4. Для распространения на Windows рядом с `.exe` должны присутствовать необходимые Qt и MinGW runtime-библиотеки.

## 10. Нефункциональные требования

### 10.1. Производительность

1. Локальные пересчеты давления, температуры и арифметики должны выполняться визуально мгновенно.
2. Перерисовка валютных полей после ввода не должна вызывать заметных задержек при стандартном количестве активных валют.
3. Сетевые задержки для валютного модуля допустимы только на этапе загрузки курсов.

### 10.2. Точность

1. Все конвертации должны использовать единый набор проверенных коэффициентов.
2. Для физических единиц не допускаются формулы, противоречащие международным справочным коэффициентам.
3. Для валют точность зависит от внешнего API и должна использоваться без несанкционированного округления до момента отображения.

### 10.3. Надежность

1. Веб-модуль валют должен корректно переживать сбой одного из поставщиков API.
2. При невозможности получить курсы пользователь должен получать явное уведомление.
3. Приложения не должны аварийно завершаться при пустом вводе или невалидном числе.

### 10.4. Совместимость

1. Pressure Converter Web должен корректно работать в актуальных версиях Chrome, Edge, Firefox и Safari.
2. Currency Converter Web должен корректно работать в браузерах с поддержкой ES Modules, Fetch API и localStorage.
3. Desktop-приложения должны работать в Windows при наличии поставленных рядом runtime-библиотек.

### 10.5. Поддерживаемость

1. Математические коэффициенты должны быть централизованы в явных структурах данных.
2. Код должен быть разделен по зонам ответственности.
3. Дублируемая логика в legacy-модулях должна по возможности заменяться общей моделью пересчета.

### 10.6. Безопасность и приватность

1. Проект не должен хранить секреты, ключи и персональные данные.
2. Единственное локальное хранилище пользовательских данных в веб-части ограничивается пользовательскими настройками интерфейса.
3. Все внешние запросы должны выполняться только к публичным API курсов валют.

## 11. Требования к сборке, запуску и публикации

### 11.1. Веб-модули

1. Корневой модуль Pressure Converter должен запускаться простым открытием `index.html` в браузере.
2. Модуль `course-convertor` должен запускаться через локальный HTTP-сервер.
3. Оба веб-модуля должны быть пригодны для публикации на GitHub Pages или другом статическом хостинге.
4. Корневой модуль использует внешние CDN-ресурсы:
   - Tailwind CSS CDN;
   - Google Fonts.

### 11.2. Desktop-модули

1. Сборка должна выполняться через Qt Creator или через `qmake` и поддерживаемый компилятор.
2. В результатах поставки должны присутствовать:
   - исполняемый файл `.exe`;
   - `Qt6Core.dll`, `Qt6Gui.dll`, `Qt6Widgets.dll` при динамической сборке;
   - `libgcc_s_seh-1.dll`, `libstdc++-6.dll`, `libwinpthread-1.dll` при MinGW-сборке;
   - платформенные плагины Qt, если модуль этого требует.
3. Файлы `.pro.user` могут присутствовать в репозитории как пользовательские артефакты Qt Creator, но не являются обязательной частью поставки.

## 13. Текущие ограничения и выявленные проблемы

### 13.1. Pressure Converter Web

- Модуль в целом реализует корректную базовую модель конвертации.
- Не используется отдельный механизм тестирования коэффициентов и форматирования.

### 13.2. Currency Converter Web

- Работа зависит от доступности внешних API.
- Отсутствует кэширование ранее загруженных курсов для офлайн-повторного использования.
- При сбое всех API пользователь получает сообщение об ошибке, но резервного локального источника нет.

### 13.3. Engineering Calculator

- Логика кнопок `del`, `%` и `AC` реализована неполно и не соответствует ожидаемому поведению классического калькулятора.
- В коде присутствуют ветки для `M+` и `M-`, но такие кнопки фактически не подтверждены UI-файлом.
- Деление на ноль явно не обрабатывается.

### 13.4. Pressure Converter Qt

- В коде присутствуют ошибки коэффициентов и направлений пересчета в нескольких ветках.
- Часть формул противоречит физически корректным соотношениям единиц.
- Модуль требует рефакторинга на единую базовую единицу Па.

### 13.5. Temperature Converter Qt

- Модуль работоспособен, но использует флаги состояния, которые можно упростить и сделать менее хрупкими.
- Поведение выбора активной шкалы желательно формализовать через единый источник состояния.

## 14. Приоритеты развития

Приоритет 1:

- поддерживать и публиковать актуальные веб-модули;
- сохранить точность конвертации давления;
- обеспечить стабильную загрузку валютных курсов и сохранение пользовательских настроек.

Приоритет 2:

- исправить критические ошибки legacy-конвертера давления;
- довести поведение инженерного калькулятора до соответствия интерфейсу.

Приоритет 3:

- унифицировать математические модели между веб- и desktop-ветками;
- при необходимости перевести legacy-логику на общую библиотеку или на новый интерфейсный слой.

## 15. Критерии приемки

### 15.1. Общие критерии

1. Репозиторий должен содержать рабочие исходники всех описанных модулей.
2. Назначение каждого модуля должно быть однозначно определено.
3. Все актуальные пользовательские сценарии должны быть подтверждены кодом и интерфейсом.

### 15.2. Pressure Converter Web

1. При вводе числа и смене единицы результаты обновляются без перезагрузки страницы.
2. Все семь единиц давления рассчитываются из одной математической модели.
3. Кнопка копирования переносит числовое значение в буфер обмена.
4. Пользователь видит уведомление после успешного копирования.

### 15.3. Currency Converter Web

1. Пользователь может вводить значение в любую активную валюту.
2. Остальные валюты пересчитываются автоматически.
3. Выбор даты влияет на загружаемые курсы.
4. Список валют и тема сохраняются между сессиями браузера.
5. При отказе основного API используется fallback-источник.

### 15.4. Legacy desktop-модули

1. Приложения запускаются на Windows при наличии приложенных runtime-библиотек.
2. Интерфейс Qt соответствует заявленным сценариям.
3. Математические результаты должны быть корректны после устранения известных дефектов.

## 16. Рекомендации по дальнейшему оформлению проекта

1. Разделить в документации актуальные продукты и архивные legacy-модули.
2. Зафиксировать единый реестр коэффициентов физических величин в одном месте.
3. Добавить smoke-тесты для конвертации давления и валютной математики.
4. Исключить из публичной документации функции, которые не доведены до рабочего состояния.
5. Рассмотреть перенос desktop-логики пересчета на общую библиотеку или полную замену legacy-модулей веб-версией.

## 17. Итог

Проект представляет собой workspace из двух актуальных веб-приложений и трех legacy desktop-приложений. Основная зрелая продуктовая ветка уже реализована в веб-формате для конвертации давления и валют, тогда как Qt-часть требует сохранения как историческая база и выборочного исправления критичных математических и логических дефектов.
