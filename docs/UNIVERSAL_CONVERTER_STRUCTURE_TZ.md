# Техническое задание на модульную структуру Universal Converter

## 1. Статус документа

- Редакция подготовлена по состоянию проекта на 17.05.2026.
- Документ фиксирует фактически реализуемую структуру корневого SPA после декомпозиции монолитного JavaScript-файла.
- Настоящая редакция заменяет прежнюю версию документа, где корневая логика считалась сосредоточенной только в `app.js`.

## 2. Объект ТЗ

Объектом настоящего ТЗ является корневое веб-приложение Universal Converter со следующими основными узлами:

- `index.html` как HTML-оболочка и точка подключения модулей;
- каталог `src` как основное местоположение прикладной логики;
- справочная документация в корне проекта.

Справочные и унаследованные части workspace:

- каталог `course-convertor` сохраняется как UX- и продуктовый референс для валютного направления;
- каталог `legacy_cpp` сохраняется как архив desktop-реализаций и предметных прототипов;
- основным пользовательским runtime-модулем считается именно корневой SPA.

## 3. Цель структурной переработки

Структурная переработка требуется для того, чтобы:

1. убрать монолитную точку сопровождения в одном JavaScript-файле;
2. развести конфигурацию, состояние, утилиты, feature-логику и DOM-оркестрацию по отдельным файлам;
3. сохранить запуск без сборки и без npm-зависимостей;
4. подготовить кодовую базу к расширению категорий и валютного сценария;
5. снизить стоимость локальных изменений и точечных проверок.

## 4. Архитектурные ограничения

Обязательные ограничения новой структуры:

1. Приложение должно оставаться статическим сайтом без сборочного шага.
2. Корневой `index.html` должен оставаться единственной HTML-точкой входа.
3. Подключение JavaScript должно выполняться через browser ES Modules.
4. Вся прикладная логика должна находиться внутри каталога `src`.
5. Конфигурация, состояние, feature-логика и утилиты не должны смешиваться в одном файле.
6. Текущая встроенная CSS-структура в `index.html` допускается до отдельного этапа выноса стилей.

## 5. Целевая файловая структура

Фактически реализованная корневая структура должна трактоваться так:

```text
/
|-- index.html
|-- README.md
|-- PROJECT_TZ.md
|-- UNIVERSAL_CONVERTER_TZ.md
|-- UNIVERSAL_CONVERTER_ROADMAP.md
|-- UNIVERSAL_CONVERTER_STRUCTURE_TZ.md
`-- src/
    |-- main.js
    |-- config/
    |   `-- app-config.js
    |-- dom/
    |   `-- elements.js
    |-- features/
    |   |-- conversion.js
    |   `-- currency.js
    |-- state/
    |   `-- app-state.js
    `-- utils/
        `-- formatters.js
```

## 6. Назначение слоев

### 6.1. Входной слой

- `index.html` содержит HTML-каркас, постоянные контейнеры интерфейса, модальные узлы и подключение модульного входа `src/main.js`.

### 6.2. Конфигурационный слой

- `src/config/app-config.js` хранит ключи localStorage, описание категорий, валютный набор и статические справочные значения.

### 6.3. DOM-слой

- `src/dom/elements.js` предоставляет централизованный доступ к корневым DOM-элементам.

### 6.4. Слой состояния

- `src/state/app-state.js` отвечает за создание клиентского состояния и восстановление базовых сохраненных настроек из браузера.

### 6.5. Feature-слой

- `src/features/conversion.js` отвечает за пересчет физических величин через адаптеры.
- `src/features/currency.js` отвечает за валютный интерфейс, локальный валютный state-flow и встроенный калькулятор.

### 6.6. Слой утилит

- `src/utils/formatters.js` содержит парсинг, форматирование, экранирование и копирование значений.

### 6.7. Оркестратор приложения

- `src/main.js` связывает все слои, инициализирует приложение, подписывает обработчики событий и управляет рендером.

## 7. Правила зависимостей

Для поддерживаемой структуры должны соблюдаться следующие правила:

1. `src/main.js` может импортировать любой нижележащий слой.
2. `features` могут импортировать `config` и `utils`, но не должны зависеть от `main.js`.
3. `state` может импортировать только `config`.
4. `utils` не должны зависеть от `state`, `features` и `main.js`.
5. Конфигурационный слой должен оставаться декларативным и не должен содержать DOM-операций.
6. Повторное форматирование и разбор пользовательского ввода не должны дублироваться в нескольких модулях.

## 8. Реализационные требования

1. Инициализация приложения должна происходить в `src/main.js` после готовности DOM.
2. Активная категория, тема и состав активных валют должны восстанавливаться из `localStorage`.
3. Категории `Давление`, `Температура` и `Валюта` должны оставаться доступными из единой навигации.
4. Валютный калькулятор должен оставаться частью валютного feature-модуля, а не общего конвертера.
5. Любое дальнейшее добавление категории должно начинаться с конфигурации и отдельного feature-слоя, а не с расширения одного монолитного файла.

## 9. Критерии приемки

Новая структура считается реализованной корректно, если:

1. `index.html` по-прежнему запускает приложение без сборки.
2. Основная бизнес-логика больше не хранится в одном корневом JavaScript-файле.
3. В проекте выделены как минимум конфигурация, состояние, утилиты и feature-модули.
4. Навигация по категориям и основные пользовательские сценарии продолжают работать после декомпозиции.
5. Перенос не требует изменения модели публикации проекта как статического сайта.

## 10. Допустимое развитие

Следующим шагом без слома этой структуры допускается:

1. вынести CSS из `index.html` в отдельный стилевой файл;
2. подключить живые курсы валют отдельным сервисным модулем;
3. выделить шаблоны рендера в самостоятельный UI-слой;
4. добавить точечные тесты на конвертацию и валютный калькулятор.
