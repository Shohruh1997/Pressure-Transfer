# Техническое задание на разработку сайта Universal Converter

## 1. Статус документа

- Документ подготовлен на основе анализа текущей структуры проекта `Pressure-Transfer`.
- Источником требований являются существующие модули: веб-конвертер давления, веб-конвертер валют, legacy-конвертер температуры на Qt и наработки по пользовательскому вводу из desktop-утилит.
- Документ описывает целевой продукт: единый сайт-универсальный конвертер, который объединяет несколько тематических конвертеров в одном интерфейсе.

## 2. Основание для разработки

Новый сайт разрабатывается на базе уже существующих решений в проекте:

1. Корневой веб-конвертер давления дает математическую модель пересчета через базовую единицу и формат карточек результатов.
2. Проект `course-convertor` дает UX-модель: выбор активных сущностей, хранение настроек, перестановку блоков, тему, работу с датой и модальное окно конфигурации.
3. Legacy-проект `Temperature-Transfer` дает набор требований для температурных формул.
4. Legacy desktop-структура показывает, какие инженерные и прикладные направления уже интересны для продукта и должны быть учтены при проектировании расширяемой архитектуры.

## 3. Назначение продукта

Необходимо создать единый современный сайт, на котором пользователь сможет:

- выбрать категорию конвертации;
- ввести значение в любой поддерживаемой единице;
- сразу получить результаты по нескольким целевым единицам;
- самостоятельно настраивать, какие единицы выводить на экран;
- менять порядок отображения результатов;
- сохранять свои настройки между сессиями;
- для валют получать актуальные и исторические курсы.

Продукт должен быть рассчитан как на обычного пользователя, так и на технического специалиста, которому важен быстрый пересчет между несколькими системами единиц без ручного выбора одной пары значений за раз.

## 4. Уникальное отличие продукта

Ключевое отличие сайта от типовых конвертеров:

1. Пользователь конвертирует не только `из одной единицы в одну`, а `из одной единицы сразу в набор выбранных единиц`.
2. Пользователь сам определяет, какие целевые единицы должны отображаться в конкретной категории.
3. Пользователь может сохранить свой набор единиц и их порядок, как это уже реализовано в валютном конвертере.
4. Для физических величин конвертация должна работать напрямую между любыми единицами через базовую математическую модель.

Пример ключевого сценария:

- пользователь вводит значение в Паскалях;
- сайт сразу показывает результат в кПа, МПа, бар, кгс/см², атм, мм рт. ст. и других выбранных единицах;
- пользователь может скрыть ненужные единицы, оставить только бар, атм и мм рт. ст., а затем сохранить этот набор.

Именно возможность мгновенного пересчета из одной единицы в несколько настраиваемых выходных единиц является главным продуктовым преимуществом.

## 5. Цель проекта

Создать единый статический веб-сайт формата SPA, который объединяет несколько конвертеров в одном интерфейсе и поддерживает расширяемую архитектуру категорий.

Цели проекта:

- объединить разрозненные конвертеры в один продукт;
- сохранить сильные стороны текущих решений;
- убрать необходимость держать отдельные приложения под каждую категорию;
- обеспечить гибкую настройку выдачи результатов;
- заложить архитектуру для добавления новых категорий без переписывания всего интерфейса.

## 6. Состав целевого продукта

В целевой продукт должны входить следующие категории конвертации.

### 6.1. MVP-категории первой версии

1. Давление.
2. Температура.
3. Валюта.

### 6.2. Категории, которые архитектура должна позволять добавить далее

1. Длина.
2. Масса.
3. Площадь.
4. Объем.
5. Скорость.
6. Энергия.
7. Мощность.
8. Время.
9. Расход.
10. Другие прикладные категории по конфигурации.

На старте не требуется реализовать все дополнительные категории, но архитектура сайта должна позволять подключать их как отдельные модули без смены базового интерфейса.

## 7. Пользователи продукта

Основные пользователи:

- инженеры, техники, студенты технических направлений;
- пользователи, которым нужен быстрый конвертер валют;
- пользователи, работающие с несколькими системами единиц одновременно;
- пользователи мобильных устройств, которым нужен компактный и быстрый интерфейс;
- администратор или владелец проекта, расширяющий список категорий и единиц.

## 8. Общая концепция интерфейса

Сайт должен быть реализован как единое SPA-приложение без серверного рендера.

На первом экране должны быть доступны:

1. Заголовок продукта.
2. Переключатель темы.
3. Навигация по категориям конвертации.
4. Основной блок ввода значения.
5. Выбор исходной единицы.
6. Список карточек результатов по выбранным целевым единицам.
7. Кнопка открытия настроек вывода.
8. Дополнительные элементы категории, если они нужны, например календарь для валют.

Интерфейс должен опираться на удачные UX-паттерны текущего `course-convertor`:

- настраиваемый список активных сущностей;
- модальное окно настройки;
- сохранение порядка;
- визуально понятный список результатов;
- адаптивная верстка;
- светлая и темная тема.

## 9. Основные пользовательские сценарии

### 9.1. Базовый сценарий для любой категории

1. Пользователь открывает сайт.
2. Пользователь выбирает категорию, например `Давление`.
3. Пользователь вводит число в основном поле.
4. Пользователь выбирает исходную единицу, например `Па`.
5. Сайт мгновенно показывает пересчет в набор выбранных целевых единиц.
6. Пользователь копирует нужный результат или меняет набор отображаемых единиц.

### 9.2. Сценарий настройки вывода

1. Пользователь открывает модальное окно настройки.
2. Пользователь отмечает чекбоксами нужные единицы для выбранной категории.
3. Пользователь убирает ненужные единицы.
4. Пользователь меняет порядок отображения карточек.
5. Сайт сохраняет набор и порядок в `localStorage`.

### 9.3. Валютный сценарий

1. Пользователь выбирает категорию `Валюта`.
2. Пользователь выбирает дату.
3. Сайт загружает актуальные или исторические курсы.
4. Пользователь вводит сумму в одной валюте.
5. Сайт показывает результаты в выбранных валютах.

### 9.4. Температурный сценарий

1. Пользователь выбирает категорию `Температура`.
2. Пользователь вводит значение, например в градусах Цельсия.
3. Сайт сразу показывает результат в Фаренгейтах, Кельвинах и других включенных единицах, если такие будут добавлены.

## 10. Функциональные требования

### 10.1. Общие требования к сайту

1. Сайт должен быть статическим и запускаться без собственного бэкенда.
2. Сайт должен работать как SPA на HTML, CSS и JavaScript.
3. Все пересчеты, кроме загрузки валютных курсов, должны выполняться на клиенте.
4. Сайт должен быть адаптивным для мобильных устройств, планшетов и десктопа.
5. Пользователь должен иметь возможность работать без регистрации.
6. Основной сценарий конвертации должен выполняться без перезагрузки страницы.
7. Все пользовательские настройки интерфейса должны сохраняться локально в браузере.

### 10.2. Требования к навигации по категориям

1. Категории должны быть доступны из единой навигации на верхнем уровне интерфейса.
2. При переключении категории сайт должен загружать соответствующий набор единиц и настроек.
3. Для каждой категории должны сохраняться собственные настройки вывода.
4. Если категория требует дополнительных параметров, они должны отображаться только внутри этой категории.

### 10.3. Требования к общему механизму конвертации

1. Пользователь должен вводить число в одном основном поле или в активной карточке текущей категории.
2. Пользователь должен выбирать исходную единицу из списка доступных единиц категории.
3. После любого изменения входного значения или исходной единицы результаты должны обновляться автоматически.
4. Система должна показывать результаты по всем активным целевым единицам, кроме выбранной исходной единицы.
5. Каждый результат должен выводиться в отдельной карточке.
6. В каждой карточке должна быть кнопка копирования значения.
7. После копирования должно отображаться уведомление об успешном действии.
8. Пользователь должен иметь возможность скрывать или показывать карточки результатов через настройки.
9. Пользователь должен иметь возможность менять порядок карточек и закреплять нужные единицы выше остальных.
10. Если введено некорректное число, интерфейс должен показывать понятное сообщение об ошибке без сбоя приложения.

### 10.4. Требования к настройке вывода единиц

Механика настройки должна быть аналогична лучшим решениям из текущего проекта `course-convertor`.

1. Для каждой категории должно существовать модальное окно настроек.
2. В модальном окне должен отображаться список всех единиц категории.
3. Единицы должны включаться и выключаться чекбоксами.
4. Пользователь должен иметь возможность оставить только нужный ему набор выходных единиц.
5. Система должна предотвращать ситуацию, когда для категории не осталось ни одной активной целевой единицы.
6. Порядок единиц должен сохраняться отдельно для каждой категории.
7. Настройки должны сохраняться в `localStorage` и восстанавливаться при следующем визите.

### 10.5. Требования к категории «Давление»

Категория должна быть реализована как один из основных модулей продукта.

Поддерживаемые единицы MVP:

- Па;
- кПа;
- МПа;
- бар;
- кгс/см²;
- атм;
- мм рт. ст.

Функциональные требования:

1. Исходной единицей по умолчанию должен быть Паскаль.
2. Математическая модель должна использовать Паскаль как базовую единицу.
3. Формула пересчета должна быть универсальной:
   - `valueInBase = inputValue * sourceFactor`;
   - `targetValue = valueInBase / targetFactor`.
4. Пользователь должен иметь возможность мгновенно получать результаты из Паскалей в бар, атм, кгс/см², мм рт. ст. и другие выбранные единицы.
5. Пользователь должен иметь возможность выбрать в качестве исходной любую поддерживаемую единицу, а не только Паскаль.
6. Приложение не должно ограничиваться одним полем `из/в`, как типичные конвертеры; оно должно показывать сразу набор карточек вывода.

Таблица коэффициентов MVP:

| Единица | Код | Коэффициент к Па |
| --- | --- | --- |
| Паскаль | `pa` | `1` |
| Килопаскаль | `kpa` | `1000` |
| Мегапаскаль | `mpa` | `1000000` |
| Бар | `bar` | `100000` |
| кгс/см² | `kgfcm2` | `98066.5` |
| Атмосфера | `atm` | `101325` |
| мм рт. ст. | `mmhg` | `133.322368` |

### 10.6. Требования к категории «Температура»

Поддерживаемые единицы MVP:

- Цельсий;
- Фаренгейт;
- Кельвин.

Функциональные требования:

1. Конвертация температуры должна поддерживать ввод в любую из трех шкал.
2. Результаты должны отображаться сразу по всем выбранным шкалам, кроме исходной.
3. Для температуры допускается отдельная математическая модель, отличная от модели простого коэффициента, так как здесь требуются формулы со смещением.
4. Внутренняя реализация должна использовать одну каноническую шкалу, например Цельсий или Кельвин.
5. Формулы должны быть физически корректны:
   - `K = C + 273.15`;
   - `F = C * 9 / 5 + 32`;
   - `C = (F - 32) * 5 / 9`;
   - `C = K - 273.15`.

### 10.7. Требования к категории «Валюта»

Категория должна использовать лучшие механики существующего `course-convertor`.

Поддерживаемые валюты MVP:

- RUB;
- UZS;
- KZT;
- USD;
- EUR;
- GBP;
- TRY;
- CNY;
- JPY;
- AED.

Функциональные требования:

1. Пользователь должен иметь возможность выбирать активные валюты для вывода.
2. Пользователь должен иметь возможность менять порядок валютных карточек.
3. Базовая внутренняя модель должна использовать USD.
4. При вводе значения в одной валюте остальные валюты должны пересчитываться автоматически.
5. Должен быть доступен выбор даты для получения исторических курсов.
6. Для текущей даты должны загружаться актуальные курсы.
7. При недоступности основного API должно использоваться резервное API.
8. Если курсы не удалось загрузить, пользователь должен получить понятное сообщение.

Требования к источникам данных:

1. Основной источник актуальных курсов: `https://open.er-api.com/v6/latest/USD`.
2. Источник исторических и резервных курсов: `@fawazahmed0/currency-api` через CDN и fallback-домен.
3. Работа валютной категории не должна требовать API-ключа.

### 10.8. Требования к будущим категориям

1. Новые категории должны подключаться как конфигурационные модули.
2. Для каждой категории должны задаваться:
   - идентификатор;
   - название;
   - тип математической модели;
   - список единиц;
   - формат отображения;
   - порядок единиц по умолчанию;
   - дополнительные параметры категории, если они нужны.
3. Добавление новой категории не должно требовать переписывания всей страницы.

## 11. Архитектурные требования

### 11.1. Общая архитектура

Сайт должен быть реализован как модульное клиентское приложение.

Рекомендуемые слои:

1. Слой конфигурации категорий.
2. Слой математических адаптеров конвертации.
3. Слой работы с внешними API.
4. Слой состояния приложения.
5. Слой DOM/UI.
6. Слой локального хранения настроек.

### 11.2. Архитектура категорий

Для каждой категории должна использоваться единая структура описания, например:

- `id`;
- `title`;
- `kind`;
- `units`;
- `defaultSourceUnit`;
- `defaultActiveUnits`;
- `formatter`;
- `parser`;
- `converterAdapter`;
- `dataProvider`, если категория зависит от API.

### 11.3. Архитектура хранения настроек

В `localStorage` должны храниться:

1. Активная категория по умолчанию.
2. Тема интерфейса.
3. Выбранные активные единицы по каждой категории.
4. Порядок карточек по каждой категории.
5. Последняя использованная исходная единица по каждой категории.
6. При необходимости последняя выбранная дата для валют.

Пример логической структуры хранения:

```json
{
  "theme": "dark",
  "activeCategory": "pressure",
  "categorySettings": {
    "pressure": {
      "sourceUnit": "pa",
      "activeUnits": ["bar", "atm", "mmhg", "kpa"],
      "order": ["bar", "atm", "mmhg", "kpa"]
    },
    "temperature": {
      "sourceUnit": "c",
      "activeUnits": ["f", "k"],
      "order": ["f", "k"]
    },
    "currency": {
      "sourceUnit": "usd",
      "activeUnits": ["rub", "eur", "uzs", "kzt"],
      "order": ["rub", "eur", "uzs", "kzt"],
      "date": "latest"
    }
  }
}
```

## 12. Требования к интерфейсу и UX

### 12.1. Общие UI-требования

1. Интерфейс должен быть современным и минималистичным.
2. С первого экрана пользователь должен понимать, что сайт умеет конвертировать несколько категорий величин.
3. Основное поле ввода должно быть визуально доминирующим.
4. Карточки результатов должны легко сканироваться взглядом.
5. На мобильных экранах интерфейс должен быть удобен для ввода одной рукой.

### 12.2. Требования к карточкам результатов

1. В карточке должно отображаться название единицы или валюты.
2. В карточке должно отображаться рассчитанное значение.
3. В карточке должна быть кнопка копирования.
4. Карточка должна быть кликабельной в части служебных действий, если они будут добавляться позднее.
5. Порядок карточек должен быть пользовательским.

### 12.3. Требования к настройкам

1. Настройки должны открываться без перехода на отдельную страницу.
2. Внутри настроек пользователь должен видеть все доступные единицы выбранной категории.
3. Изменения должны применяться сразу после выбора.
4. Пользователь должен видеть, какие единицы уже активны.

### 12.4. Темизация

1. Сайт должен поддерживать светлую и темную тему.
2. При первом открытии тема должна определяться по системным настройкам пользователя.
3. После ручного переключения тема должна сохраняться.

## 13. Технологические требования

1. Реализация должна использовать HTML5, CSS3 и JavaScript ES6+.
2. Допускается реализация без сборщика, если архитектура остается поддерживаемой.
3. Код должен быть модульным.
4. Сайт должен быть пригоден для публикации на статическом хостинге.
5. Не допускается обязательная серверная часть для работы базовых категорий.
6. Внешние API допускаются только для категорий, которым действительно нужны внешние данные, например валюты.

## 14. Нефункциональные требования

### 14.1. Производительность

1. Пересчет локальных физических величин должен происходить визуально мгновенно.
2. Переключение категории не должно вызывать заметной задержки.
3. Интерфейс должен оставаться отзывчивым при работе с 10-20 карточками результатов.

### 14.2. Точность

1. Для категорий коэффициентного типа должны использоваться централизованные и проверяемые коэффициенты.
2. Для температур должны использоваться строгие формулы.
3. Для валют результат зависит от внешних API, но после загрузки курсов пересчет должен быть математически корректен.

### 14.3. Надежность

1. Ошибка загрузки валютных курсов не должна ломать остальные категории.
2. Ошибки ввода должны обрабатываться безопасно.
3. Пустой ввод не должен вызывать падение интерфейса.

### 14.4. Совместимость

1. Сайт должен работать в актуальных версиях Chrome, Edge, Firefox и Safari.
2. Сайт должен быть пригоден для использования на Android и iPhone.
3. Для локального запуска валютной части должен допускаться запуск через локальный HTTP-сервер.

### 14.5. Поддерживаемость

1. Новые единицы должны добавляться через конфигурацию, а не через копипасту логики.
2. Новые категории должны подключаться отдельным модулем.
3. Коэффициенты и формулы должны храниться централизованно.

## 15. Ограничения проекта

1. В первой версии не требуется пользовательская регистрация.
2. В первой версии не требуется синхронизация настроек между устройствами.
3. В первой версии не требуется собственный сервер курсов валют.
4. В первой версии не требуется полноценный инженерный калькулятор внутри сайта.

При этом опыт старого `Engineering-Calculator` допускается использовать для улучшения ввода чисел, форматирования и поведения клавиатуры, но сам калькулятор не является обязательной частью MVP универсального конвертера.

## 16. Состав MVP

В MVP обязательно должны войти:

1. Единый сайт с тремя категориями: давление, температура, валюта.
2. Переключение категорий без перезагрузки страницы.
3. Настраиваемый список целевых единиц/валют.
4. Сохранение настроек в `localStorage`.
5. Темная и светлая тема.
6. Копирование результата.
7. Исторические курсы для валют.
8. Мгновенный пересчет в несколько карточек результатов.

## 17. Состав второй очереди

Во вторую очередь рекомендуется включить:

1. Дополнительные категории: длина, масса, объем, площадь.
2. Поиск по единицам в модальном окне настроек.
3. Избранные категории и избранные единицы.
4. Кэш последних валютных курсов.
5. Улучшенный режим клавиатурного ввода.
6. Локализацию интерфейса на другие языки.

## 18. Критерии приемки

Продукт считается соответствующим ТЗ, если выполняются следующие условия.

### 18.1. Общие критерии

1. Сайт открывается как единое приложение.
2. Пользователь может переключаться между категориями без перезагрузки.
3. Для каждой категории доступны собственные настройки вывода.

### 18.2. Давление

1. Пользователь вводит значение в Паскалях и сразу видит результаты в бар, атм, кгс/см², мм рт. ст. и других выбранных единицах.
2. Пользователь может изменить исходную единицу на любую другую поддерживаемую единицу.
3. Пользователь может скрыть лишние единицы и сохранить свой набор.

### 18.3. Температура

1. Пользователь вводит температуру в любой из трех шкал.
2. Остальные шкалы пересчитываются автоматически.
3. Формулы дают корректный результат.

### 18.4. Валюта

1. Пользователь выбирает дату и получает курсы на эту дату.
2. Пересчет между активными валютами работает автоматически.
3. Настройки валют сохраняются между сессиями.

### 18.5. UX

1. Пользователь может менять порядок карточек результатов.
2. Пользователь может копировать результат из карточки.
3. Сайт сохраняет тему и настройки вывода.

## 19. Рекомендации по реализации

1. Использовать единый реестр категорий с конфигурацией единиц.
2. Реализовать универсальный движок карточек результатов, чтобы переиспользовать его между давлением, температурой и валютами.
3. Для категорий коэффициентного типа использовать общую базовую формулу через базовую единицу.
4. Для категорий особого типа, например температуры и валют, использовать адаптеры со своей математикой и загрузкой данных.
5. Перенести лучшие UX-решения из `course-convertor` в общий продукт, а лучшие вычислительные подходы из текущего конвертера давления — в физические категории.

## 20. Итоговая формулировка продукта

Итоговый продукт должен представлять собой единый универсальный сайт-конвертер, который объединяет валюты, давление, температуру и другие категории в одном интерфейсе. Главная ценность продукта — не просто перевод `из одной единицы в одну`, а мгновенный показ нескольких нужных пользователю результатов с возможностью настроить набор и порядок отображаемых единиц под свои задачи.
